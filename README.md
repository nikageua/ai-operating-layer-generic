# AI Operating Layer

A generic AI operating system for multi-repository, multi-framework feature development.

## Purpose

- Give AI agents stable, structured context before they act
- Drive features through spec → epic → task → sub-agent execution
- Tune sub-agent harnesses empirically, not by guessing upfront
- Prevent scope creep and uncoordinated changes across repositories

## The Flow

```
Feature description (human, even one sentence)
  ↓
Spec co-authoring         .ai/specs/          draft → review → approved → implemented
  ↓
Architecture decisions    .ai/architecture/   engine decisions; spike tasks if needed
  ↓
Epic breakdown            .ai/epics/          MVP declared per epic; backlog → done
  ↓
Task breakdown            .ai/tasks/          SMART tasks; one profile per task
  ↓
Sub-agent execution       .ai/agents/         profile-tuned harness per framework/repo
  ↓
Harness optimization      .ai/metaharness/    auto → proposal → stable (tiered)
```

## Layers

| Folder | Purpose |
|--------|---------|
| `specs/` | Spec-driven development. Entry point for every feature. |
| `epics/` | Large feature chunks. MVP scope lives here, not in architecture. |
| `tasks/` | SMART atomic tasks assigned to sub-agents. |
| `agents/` | Sub-agent profiles (one per framework/repo combo). Includes bootstrap protocol. |
| `metaharness/` | Harness optimization rules. Profiles evolve empirically via tiered updates. |
| `architecture/` | Engine decisions and cross-cutting technical constraints. |
| `brain/` | Current working state: focus, decisions, next steps, open questions. |
| `memory/` | Durable patterns and lessons across sessions. |
| `agent/` | Main agent rules and operating protocol (AGENTS.md). |
| `project/` | Stable project context and current status. |

## Key Design Decisions

**Specs before code.** Epics and tasks cannot be created without an approved spec.
This prevents building the wrong thing.

**MVP per epic, not global.** Each epic declares its own MVP scope.
This scales across multiple features and repositories.

**Harness bootstrapped, not pre-written.** Sub-agent profiles start from a thin
bootstrap generated by inspecting the target repo. They improve with every task run
via the metaharness optimization loop.

**Tiered harness updates.** The metaharness loop runs automatically (Tier 1).
Promoting a result back into a profile requires human approval (Tier 2).
Changing acceptance criteria is always manual (Tier 3).

**Two agent roles.** The orchestrator manages specs, epics, and tasks.
Sub-agents execute one task at a time using their declared profile.

# Getting Started

## For a New Project

### 1. Copy the template
Copy the entire `.ai/` folder from this template into your project root.

```bash
cp -r ai-operating-layer-generic/.ai /path/to/your/project/
```

### 2. Copy the root prompt file
```bash
cp ai-operating-layer-generic/INIT_PROMPT.md /path/to/your/project/
```

### 3. Customize for your project

#### `.ai/project/PROJECT_CONTEXT.md`
Fill in:
- What repositories are involved (e.g. frontend-repo, backend-repo)
- What frameworks you're using (e.g. Angular 20, .NET 8)
- What the business/product goal is
- Any constraints or limitations

#### `.ai/project/CURRENT_STATUS.md`
Fill in:
- Current state of the project
- What is working, what is blocked
- Any known issues

#### `.ai/architecture/ARCHITECTURE_DECISIONS.md`
Replace the placeholder with your actual tech stack decisions:
- Primary languages and frameworks
- Database choices
- Deployment architecture
- Any critical constraints (multi-arch, multi-database support, etc.)

#### `.ai/epics/EPIC_SELECTION_RULES.md` (if needed)
Update the selection rules if your project has special priorities or constraints.

#### `.ai/agents/profiles/`
**Important:** If you use frameworks other than Angular or .NET, you must generate new profiles.
Run the harness bootstrap protocol:
1. When a task declares a profile that doesn't exist, the orchestrator agent will trigger bootstrap
2. Bootstrap inspects your target repo and auto-generates a profile
3. A harness validation task confirms the profile works
4. The profile moves from `bootstrap` → `draft` → `validated` → `stable`

#### `.ai/memory/`
Add your domain rules, naming conventions, and technical patterns as you learn them:
- `DOMAIN_RULES.md` — Product/business rules that must always hold
- `CONTENT_CONVENTIONS.md` — Naming and file structure conventions
- `TECHNICAL_PATTERNS.md` — Code patterns and architectural preferences
- `KNOWN_PITFALLS.md` — Failure modes to avoid

### 4. Read the governance protocols
Before the first agent session, read these three files in order:
1. `.ai/agent/AGENTS.md` — The complete operating protocol
2. `.ai/brain/BRAIN_UPDATE_PROTOCOL.md` — How to maintain brain files
3. `.ai/memory/MEMORY_GOVERNANCE.md` — How to evolve memory files

## Starting Your First Feature

### Step 1: Provide a feature description
Give the orchestrator agent a human-language description of what you want to build.
Even one sentence is enough:

> "I want to add album management — users can create, edit, and delete photo albums."

### Step 2: Co-author the spec
The agent will ask clarifying questions and help you fill out a spec template:
- Problem statement (what problem this solves)
- User stories (as a X, I want Y so that Z)
- Acceptance criteria (given X, when Y, then Z)
- Affected repositories and frameworks
- Open questions and spike tasks (if needed)

### Step 3: Approve the spec
Once the spec is complete and questions are answered, the human approves it.
The spec moves from `draft` → `review` → `approved`.

### Step 4: Orchestrator breaks it into epics
The agent proposes epic breakdown:
- Each epic maps to one or more repositories/frameworks
- Each epic declares MVP scope (MVP vs. Post-MVP vs. Nice-to-have)
- Each epic gets a `Parent Spec` reference

### Step 5: Break epics into SMART tasks
When an epic is activated (`in-progress`), the orchestrator breaks it into SMART tasks:
- **Specific** — exactly one code change or deliverable
- **Measurable** — clear acceptance criteria
- **Achievable** — can be done in one session
- **Relevant** — directly advances the epic
- **Time-bounded** — scoped small

Each task declares which sub-agent profile it needs (e.g. `angular-frontend`, `dotnet-api`, `generic`).

### Step 6: Sub-agents execute tasks
For each task:
1. Sub-agent reads its profile (`.ai/agents/profiles/[profile-id].md`)
2. Confirms the harness passes (bootstrap, validate, test scripts)
3. Implements the task within stated scope
4. Confirms the harness still passes
5. Proposes an AI layer update

### Step 7: Harness improves automatically
After each task, the metaharness loop runs automatically:
- **Tier 1 (automatic):** metaharness evaluates candidate improvements
- **Tier 2 (proposal):** orchestrator proposes promoting the best candidate back to the profile
- **Tier 3 (manual):** human approves changes to acceptance criteria or profile status

## Key Commands / Actions

### Starting a spec
```
Human → Orchestrator agent (reads INIT_PROMPT.md)
Input: Feature description (one sentence)
Output: Spec file in .ai/specs/draft/
```

### Approving a spec
```
Proposed AI layer update:

Specs
- move SPEC_001 from draft → review

Approve? (Yes / No)
```

### Activating an epic
```
Proposed next epic:

- recommended epic: EPIC_001
- reason: [rationale]

Proposed AI layer update:

Epics
- move EPIC_001 from ready → in-progress

Approve? (Yes / No)
```

### Activating a task
```
Proposed next task:

- recommended task: TASK_001 (epic: EPIC_001)
- profile: dotnet-api
- reason: [rationale]

Proposed AI layer update:

Tasks
- move TASK_001 from ready → in-progress

Approve? (Yes / No)
```

### Proposing harness improvements (after task completion)
```
Proposed harness update:

Profile: dotnet-api
Change: Update Key Instructions to include dependency injection pattern
Evidence: TASK_001 discovered inconsistent DI usage
Diff:
  - old: [old instruction]
  + new: [new instruction]

Approve harness update? (Yes / No)
```

## Troubleshooting

### "Profile does not exist"
**Solution:** Trigger harness bootstrap. The orchestrator will:
1. Inspect the target repository
2. Auto-generate a profile in `.ai/agents/profiles/[id].md`
3. Create a harness validation task
4. After validation passes, the real task can proceed

### "Brain files are out of sync"
**Solution:** Run the consistency checklist at the start of the session (see `.ai/brain/BRAIN_UPDATE_PROTOCOL.md`).
Verify that `SESSION_HANDOFF.md`, `CURRENT_FOCUS.md`, and `NEXT_STEPS.md` match the actual task status.

### "A rule needs to change"
**Solution:** Propose a rule change using the format in `.ai/architecture/RULES_EVOLUTION.md`.
Soft-locked rules can be changed via proposal. Tier 3 rules require explicit human instruction.

### "I'm adding a new framework"
**Solution:** Create a new task and declare a new profile ID.
Bootstrap will auto-generate the profile by inspecting the target repo.
A harness validation task confirms it works before real implementation starts.
